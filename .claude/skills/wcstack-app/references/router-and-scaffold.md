# wcstack router / autoloader / app scaffold reference

Sources: `packages/router/README.ja.md`, `packages/autoloader/README.ja.md`, root `README.md`, `examples/README.ja.md`, `examples/router-spa/` (index.html / server.js / README.ja.md), `packages/router/src/`. All verified against the actual files.

## 1. Minimal SPA scaffold

### CDN loading (one package per line, `/auto` entry)

```html
<script type="module" src="https://esm.run/@wcstack/fetch/auto"></script>
<script type="module" src="https://esm.run/@wcstack/router/auto"></script>
<script type="module" src="https://esm.run/@wcstack/state/auto"></script>
```

`/auto` is a zero-config bootstrap that only performs registration. Best practice is to list I/O node packages before state (not a hard requirement, since state defers via whenDefined).

### Basic router structure

**A `<template>` is required directly under `<wcs-router>`**. Place `<wcs-route>` elements inside it. The render target is `<wcs-outlet>` (auto-generated by the router if not written in the HTML).

```html
<wcs-router>
  <template>
    <wcs-route path="/">
      <wcs-head><title>Home</title></wcs-head>
      <app-home></app-home>
    </wcs-route>
    <wcs-route path="/about">
      <about-page></about-page>
    </wcs-route>
    <wcs-route fallback>
      <error-404></error-404>
    </wcs-route>
  </template>
</wcs-router>
<wcs-outlet></wcs-outlet>
```

- To display a component, **write it directly as a child of `<wcs-route>`**. On match, the children are stamped into `<wcs-outlet>`.
- Static HTML (without `data-wcs`) may also be written directly inside a route.

### `<wcs-route>` attribute list

| Attribute | Description |
|------|------|
| `path` | Absolute path starting with `/` at the top level; relative path when nested. Parameters are `:name`. Catch-all is `*`. Relative paths are not allowed at the top level |
| `index` | Inherits the parent path (matches the parent path itself) |
| `fallback` | Displayed when no route matches |
| `fullpath` | Full path including ancestors (read-only) |
| `name` | For identification |
| `guard` | Enables the guard. The value is the absolute redirect path used when the guard cancels |

Properties: `params` (string parameters) / `typedParams` (type-converted) / `guardHandler`.

### Matching priority

1. More segments → 2. more static segments (`"users"` > `":id"` > `"*"`) → 3. definition order. `/products` and `/products/` are treated as identical.

## 2. Typed path parameters

Syntax: `:paramName(typeName)`

```html
<wcs-route path="/users/:userId(int)"><user-detail data-bind="props"></user-detail></wcs-route>
<wcs-route path="/posts/:date(isoDate)/:slug(slug)"><post-detail data-bind="props"></post-detail></wcs-route>
```

| Type name | Description | Converted type |
|------|------|------|
| `int` | Integer | `number` |
| `float` | Floating-point number | `number` |
| `bool` | `true`/`false`/`0`/`1` | `boolean` |
| `uuid` | UUID v1-5 | `string` |
| `slug` | Lowercase alphanumerics and hyphens | `string` |
| `isoDate` | ISO 8601 date | `Date` |
| `any` | Any string (default) | `string` |

A value that fails the type check **does not match that route** (no error; it falls through to other routes or the fallback route). Unknown type names fall back to `any`.

### How to receive parameters

**(a) From JS**: `route.params.userId` (string) / `route.typedParams.userId` (type-converted).

**(b) `data-bind` auto-binding** — automatically injects parameters into elements inside the route:

| `data-bind` value | Behavior |
|------|------|
| `"props"` | Merged into `element.props` |
| `"states"` | Merged into `element.states` |
| `"attr"` | Set via `setAttribute()` |
| `""` (empty string) | Direct property (`element.userId = value`) |

Parameters are assigned before `connectedCallback` fires. For undefined custom elements, assignment is deferred until `customElements.whenDefined()` resolves (compatible with the autoloader).

**(c) From state (via wc-bindable)** — derive with getters from the `path` output of `<wcs-router>` (see §5 and §7).

## 3. Nested layouts and `<wcs-head>`

### `<wcs-layout>`

Loads a template, inserts its children into `<slot>`, and writes the result out to `<wcs-layout-outlet>`.

| Attribute | Description |
|------|------|
| `layout` | id of the `<template>` used as the layout template |
| `src` | URL of an external file template |
| `name` | Identifier name |
| `enable-shadow-root` / `disable-shadow-root` | Toggles Shadow/Light DOM for the outlet |

### Real example mixing routes and layouts

```html
<wcs-router>
  <template>
    <wcs-route path="/">
      <wcs-layout layout="main-layout">
        <main-header slot="header"></main-header>
        <main-body>
          <wcs-route index>
            <wcs-head><title>Main Page</title></wcs-head>
            <main-dashboard></main-dashboard>
          </wcs-route>
          <wcs-route path="products">          <!-- nested routes use relative paths -->
            <wcs-route index><product-list></product-list></wcs-route>
            <wcs-route path=":productId"><product-item data-bind="props"></product-item></wcs-route>
          </wcs-route>
        </main-body>
      </wcs-layout>
    </wcs-route>
    <wcs-route fallback><error-404></error-404></wcs-route>
  </template>
</wcs-router>
<wcs-outlet></wcs-outlet>

<template id="main-layout">
  <section><h1> Main </h1><slot name="header"></slot></section>
  <section><slot></slot></section>   <!-- route content goes into the default slot -->
</template>
```

### Light DOM limitation

With `disable-shadow-root`, slot substitution applies **only to direct children of `<wcs-layout>`**. Elements with a `slot` attribute inside a `<wcs-route>` are not slotted.

### `<wcs-head>`

Manages the document `<head>` per route. Stack-based: the last one connected wins. Supported elements: `<title>` `<meta>` `<link>` `<base>` `<script>` `<style>`. When all `<wcs-head>` elements disconnect, the initial state is restored. `<meta>` is identified by `name`/`property`/`http-equiv`; `<link>` by `rel`/`href`.

## 4. Guards, basename, fallback

### Route guards

```html
<wcs-route path="/dashboard" guard="/login">
  <wcs-guard-handler>
    <script type="module">
      export default function(toPath, fromPath) {
        return document.cookie.includes('session=');   // boolean | Promise<boolean>
      }
    </script>
  </wcs-guard-handler>
  <dashboard-page></dashboard-page>
</wcs-route>
```

`false` cancels → navigates to the path in the `guard` attribute. `<wcs-guard-handler>` is removed from the DOM after parsing. Ignored if placed outside a `<wcs-route>`. Can also be set from JS via `route.guardHandler = fn`.

### basename

```html
<wcs-router basename="/app"> ... </wcs-router>
```

Resolution order: (1) `basename` attribute → (2) derived from `<base href>` → (3) empty string. Normalization: prepend leading `/`, collapse consecutive slashes, strip trailing `/`, strip trailing `*.html` (`/app/index.html` → `/app`). Multiple routers can coexist in the same document as long as their basenames differ.

### Fallback route / catch-all

- `<wcs-route fallback>` — for 404.
- A trailing `*` in `path` matches the entire remaining path; retrieve it via `params['*']`; lowest priority.

## 5. Navigation

### Declarative links: `<wcs-link>` (use this, not a raw `<a>`)

```html
<wcs-link to="/">Products</wcs-link>
<wcs-link to="/about">About</wcs-link>
```

- Converted to an `<a>`. If `to` starts with `/`, it is a route path (basename auto-prepended); otherwise it is treated as an external URL.
- The `active` CSS class is automatically applied on current-location match: `a.active { font-weight: bold; }`

### Programmatic navigation

**(a) JS API**: the router element's `async navigate(path: string): Promise<void>` (Navigation API, with pushState fallback when unavailable).

**(b) From state** — the wc-bindable surface of `<wcs-router>` is `path` (output-only) and `navigateUrl` (both input/output):

```html
<wcs-router data-wcs="path: path; navigateUrl: navigateUrl">
```

```javascript
export default {
  path: "",          // router → state (current path flows in)
  navigateUrl: null, // state → router (assign to navigate)
  openProduct() { this.navigateUrl = "/products/" + this["products.*.id"]; },
  goToProducts() { this.navigateUrl = "/"; },
};
```

- `navigateUrl` is **self-resetting**: the router sets it back to `null` when navigation completes, so re-assigning the same path navigates again.
- `path` is output-only, so never write it back from state (the router is the authority. The current value is read when the binding is established, so deep links are never missed).

## 6. autoloader

### Import Map + element placement

```html
<script type="importmap">
  {
    "imports": {
      "@components/ui/": "./components/ui/",
      "@components/app/": "./components/app/"
    }
  }
</script>
<script type="module" src="https://esm.run/@wcstack/autoloader/auto"></script>
<body>
  <wcs-autoloader></wcs-autoloader>   <!-- Triggers the load lifecycle. Required -->
  <ui-button></ui-button>    <!-- auto-imports ./components/ui/button.js -->
  <app-header></app-header>  <!-- auto-imports ./components/app/header.js -->
</body>
```

### Resolution rules

**Lazy loading (namespace)** — key ends with `/`: `"@components/<prefix>[|<loader>]/": "<path>"`
- The prefix part of the tag name matches and the remainder becomes the file name: `@components/ui/` + `<ui-button>` → `./components/ui/button.js`
- Slashes inside the prefix are converted to dashes. Loader specification: `"@components/ui|lit/": "./ui/"` (default `vanilla`)

**Eager loading** — key does not end with `/`: `"@components/my-button": "./my-button.js"`, `"@components/fancy-input|vanilla,input": "./fancy-input.js"` (when extends is omitted, it is auto-detected from the prototype)

**Component requirements** (vanilla loader): `.js` extension + `default` export of the custom element class:

```javascript
// components/ui/button.js
export default class UiButton extends HTMLElement {
  constructor() {
    super();
    this.attachShadow({ mode: 'open' }).innerHTML = '<button><slot></slot></button>';
  }
}
```

- The `is` attribute (customized built-ins) is also auto-detected (the autoloader defines with `{ extends }`).
- Components that fail to load are not retried (failure detection is the responsibility of `@wcstack/defined`). Dynamically added elements are also detected via MutationObserver.

## 7. Full index.html structure of the real demo (actual examples/router-spa skeleton)

```html
<!DOCTYPE html>
<html lang="en">
<head>
  <meta charset="UTF-8">
  <meta name="viewport" content="width=device-width, initial-scale=1.0">
  <!-- Required for deep links: without it, loading /products/3 directly makes that path the basename -->
  <base href="/">
  <title>wcstack: router + state + fetch demo</title>
  <style>/* style active links via a.active */</style>
  <script type="module" src="https://esm.run/@wcstack/state/auto"></script>
  <script type="module" src="https://esm.run/@wcstack/fetch/auto"></script>
  <script type="module" src="https://esm.run/@wcstack/router/auto"></script>
</head>
<body>

<nav aria-label="Main">
  <wcs-link to="/">Products</wcs-link>
  <wcs-link to="/about">About</wcs-link>
</nav>

<wcs-state>
  <script type="module">
    export default {
      path: "",
      navigateUrl: null,
      productsFetch: { value: null, loading: false, error: null, status: 0 },
      productFetch: { value: null, loading: false, error: null, status: 0 },
      get productId() {
        const m = this.path.match(/^\/products\/(\d+)$/);
        return m ? Number(m[1]) : null;
      },
      get isList() { return this.path === "/" || this.path === ""; },
      get isDetail() { return this.productId !== null; },
      get "productFetch.url"() {
        return this.productId ? "/api/products/" + this.productId : undefined;
      },
      get products() { return this["productsFetch.value"] ?? []; },
      openProduct() { this.navigateUrl = "/products/" + this["products.*.id"]; },
      goToProducts() { this.navigateUrl = "/"; },
    };
  </script>
</wcs-state>

<!-- Router: owns the URL, history, page <title>, and static pages -->
<wcs-router data-wcs="path: path; navigateUrl: navigateUrl">
  <template>
    <wcs-route path="/">
      <wcs-head><title>Products</title></wcs-head>
    </wcs-route>
    <wcs-route path="/products/:productId(int)">
      <wcs-head><title>Product detail</title></wcs-head>
    </wcs-route>
    <wcs-route path="/about">
      <wcs-head><title>About</title></wcs-head>
      <section><!-- write static content directly (no data-wcs) --></section>
    </wcs-route>
    <wcs-route fallback>
      <wcs-head><title>Not found</title></wcs-head>
      <section>
        <h2>404</h2>
        <p><wcs-link to="/">Back</wcs-link></p>
      </section>
    </wcs-route>
  </template>
</wcs-router>
<wcs-outlet></wcs-outlet>

<!-- Headless fetch nodes (spread all outputs into state slots) -->
<wcs-fetch url="/api/products" data-wcs="...: productsFetch"></wcs-fetch>
<wcs-fetch data-wcs="...: productFetch"></wcs-fetch>

<!-- Data-bound pages live under state-managed <template data-wcs="if:"> (directly under body) -->
<template data-wcs="if: isList">
  <section aria-label="Product list">
    <ul>
      <template data-wcs="for: products">
        <li>
          <button type="button" data-wcs="onclick: openProduct; attr.aria-label: .name">
            <span data-wcs="textContent: .name"></span>
            <span>¥<span data-wcs="textContent: .price|locale('ja-JP')"></span></span>
          </button>
        </li>
      </template>
    </ul>
  </section>
</template>

<template data-wcs="if: isDetail">
  <section aria-label="Product detail">
    <button type="button" data-wcs="onclick: goToProducts">&larr; Back</button>
    <h2 data-wcs="textContent: productFetch.value.name"></h2>
  </section>
</template>

</body>
</html>
```

### Server-side SPA fallback (actual examples/router-spa/server.js)

```javascript
// SPA fallback: every non-API, extensionless GET returns index.html
if (!url.pathname.startsWith("/api/") && extname(url.pathname) === "") {
  const html = await readFile(join(__dirname, "index.html"));
  res.writeHead(200, { "Content-Type": "text/html; charset=utf-8" });
  res.end(html);
  return true;
}
```

## 8. How to put your own custom elements on a route

- Write the tag directly inside the route and receive parameters via `data-bind` (§2).
- **Defining is outside the router's responsibility** — the autoloader (§6) or a manual `customElements.define()` is required separately.
- Alternative pattern (proven in router-spa): create no custom elements; put only `<wcs-head>` + static content in the routes, and switch the data-bound page DOM via `path`-derived getters in `<template data-wcs="if: ...">` directly under body (because of the constraint in §9-3, this pattern is the default answer when combining with state).

## 9. Pitfall checklist

1. **Deep links break without `<base href="/">`** (the basename is mis-derived from `document.baseURI`).
2. **The server needs a SPA fallback** — without it, reloads and direct links 404.
3. **Writing `data-wcs` on nodes stamped by the router is not observed by state** — state only collects the DOM present at bind time. Put data bindings in `<template data-wcs="if:">` directly under body.
4. Script order — best practice is to list I/O node packages before `@wcstack/state`.
5. In Light DOM layouts, elements with a `slot` attribute are only effective as direct children of `<wcs-layout>`.
6. A type mismatch is not an error but "no match" — `/products/abc` passes through `:productId(int)` and falls to the fallback.
7. Relative paths cannot be written on top-level routes. Nested routes use relative paths.
8. `undefined` is never written to an element (write-skip) — if a getter returns `undefined`, `<wcs-fetch>` stays silent. The url same-value guard prevents re-fetching the same URL.
9. Autoloader load failures are not retried.
