@wcstack/fetch

Pagination, five ways

The same paginated member list — 200 records, 12 per page, server-side latency — built five times against one shared server. The data, the markup and the styling are identical; only the front-end approach changes. All five drive the same headless <wcs-fetch> node through the wc-bindable protocol — each via a different binding layer — so the fetch, the abort and the stale-response protection live in the element, and the only thing that varies is how each paradigm subscribes to it. Each demo surfaces the node's state machine (page → state → HTTP status) to make that visible: you don't orchestrate the async flow, you just read which state the node is in. Compare how each binds the page state, the re-fetch, and the cancellation of a superseded request.

@wcstack/state

wcstack / state

Declarative <wcs-fetch> + data-wcs bindings. No JavaScript glue — the URL getter drives the fetch.

 @wcstack/signals

wcstack / signals

Fine-grained signals: bindNode() adapts <wcs-fetch> into signals, and the element cancels the previous page on each change.

 Vanilla JS

Vanilla

Hand-built DOM, driven by a headless <wcs-fetch> node that @wc-bindable/core's bind() streams into a plain state object. The framework-free baseline.

 React

React

useState + a headless <wcs-fetch> node bound with @wc-bindable/react's useWcBindable — the element owns the fetch + abort.

Vue

Running the demos

state / signals / vanilla are buildless — this page and all three are served by the shared hub (every dependency, including @wcstack/signals, loads from the CDN, so there is nothing to build):

node packages/fetch/examples/pagination/shared/server.js
# open http://localhost:3400

React / Vue use Vite, so each builds and serves itself (reusing the same /api/items):

cd packages/fetch/examples/pagination/react && npm install && npm run start   # :3404
cd packages/fetch/examples/pagination/vue   && npm install && npm run start   # :3405
